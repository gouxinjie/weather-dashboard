# AGENTS.md

## 项目说明

- 项目中文名：城市环境与天气大屏
- 项目英文名：`City Weather Dashboard`
- 项目类型：单城市天气与环境可视化大屏
- 主数据源：和风天气
- 核心目标：突出天气主模块，展示实时天气、24 小时趋势、7 天天气预报、空气质量、灾害预警、生活指数、周统计与月统计

## 产品范围

- 主页面：城市环境与天气大屏首页
- 城市选择：使用抽屉，不单独做页面
- 统计详情页：展示近 30 天温度、降水、AQI 与统计说明
- 首次进入页面优先定位当前城市，失败时回退默认城市
- 当前月统计按“截至最新采样日”展示，必须标注样本期

## 技术栈要求

- 前端必须使用 `React + Vite + TypeScript`
- 图表使用 `ECharts`
- 样式必须使用 `SCSS`
- 路由使用 `React Router`
- 状态管理使用 `Zustand`
- 后端必须使用 `Node.js + TypeScript`
- 后端框架使用 `Express` 或 `Fastify`
- 数据库使用 `SQLite`

## 代码强约束

- 必须使用 TypeScript
- 禁止使用 `any`
- 所有函数、变量、接口响应必须定义明确类型
- 使用函数组件 + Hooks
- 禁止 class component
- 遵循 ESLint 规则（`@typescript-eslint` / `react/recommended`）
- 禁止使用 `eslint-disable`
- 必须移除未使用变量、未使用函数、未使用 import、未使用样式
- 不允许修改无关文件

## 注释要求

- 所有代码注释必须使用中文
- 每个组件文件必须包含头部注释：

```ts
/**
 * @component 组件名称
 * @description 组件描述
 * @author
 * @created
 * @updated
 */
```

- Props 或接口定义必须说明：
  - 类型
  - 含义
  - 是否必填
  - 默认值
- 工具函数、API 封装函数必须包含功能、参数、返回值、异常说明
- 数据库设计与 SQL 相关说明必须使用中文注释

## API 规范

- 所有前后端业务请求必须包含 `userId`
- 当前项目虽然是单用户模式，但接口层仍保留 `userId` 字段
- 如未提供真实用户体系，必须使用明确的系统用户配置，不允许前端随意伪造业务含义
- 所有请求必须处理 `loading` 和 `error`
- 不允许假设接口结构
- 请求必须集中封装在 `src/utils/request.ts` 或按模块拆分封装
- 页面中禁止散写第三方请求逻辑
- 前端不直接调用和风天气，必须由后端转发与聚合

## 后端开发规范

### 分层要求

- 后端必须按职责分层，至少拆分为：
  - `routes`
  - `controllers`
  - `services`
  - `repositories` 或 `db`
  - `types`
  - `utils`
- `controller` 只负责参数接收、调用 service、返回统一响应
- `service` 负责业务编排、第三方聚合、统计计算
- `repository` 或数据库层只负责数据访问
- 禁止在 `controller` 中直接写 SQL
- 禁止在 `controller` 中直接调用和风天气

### 接口实现要求

- 每个接口必须定义请求类型和响应类型
- 每个接口必须校验参数
- 参数不合法时必须返回统一错误结构
- 所有接口必须返回中文错误信息
- 聚合接口必须明确字段来源，禁止隐式拼装不清晰的数据结构
- 首页聚合接口、统计接口、定位解析接口必须单独实现，不允许把所有逻辑堆在一个文件中

### 第三方调用要求

- 所有和风天气请求必须集中封装
- 不允许在多个业务文件中散写重复的第三方请求逻辑
- 必须处理第三方超时、失败、限流和空数据情况
- 第三方返回字段必须经过转换后再返回给前端
- 不允许把第三方原始响应直接透传给前端

### 数据库要求

- SQLite 访问必须通过统一数据库模块或仓储层管理
- 所有查询必须使用参数化查询
- 所有写入必须明确字段，不允许隐式写整对象
- 快照表、聚合表、配置表必须分开管理
- 聚合统计必须基于已落库数据生成，不允许直接依赖前端实时计算
- 涉及重算的周统计和月统计，必须使用可重复执行的重算逻辑

### 缓存与同步要求

- 必须对第三方天气数据做缓存，避免无意义重复请求
- 缓存策略必须按接口类型区分，例如实时天气、AQI、预警、预报分别设置过期时间
- 同步任务和接口查询逻辑必须分开
- 接口读取优先使用缓存或本地快照
- 若缓存失效且第三方调用失败，必须有兜底策略

### 错误与日志要求

- 后端必须统一处理异常
- 不允许直接把原始异常栈返回给前端
- 第三方请求失败必须记录日志
- 数据库写入失败必须记录日志
- 日志中禁止输出敏感信息
- 生产环境日志必须可控，不允许无边界打印

### 安全要求

- 所有密钥必须从环境变量读取
- 不允许把和风天气 Key 写入源码
- 所有 POST 接口必须校验 CSRF 令牌
- 所有输入参数必须做校验和必要转义
- 不允许信任前端传入的业务关键字段

## 当前项目接口边界

- 主数据源固定为和风天气
- 对前端公开的业务接口统一以 `locationId` 为主键
- 浏览器定位得到的 `lat/lon` 只能传给自有后端接口，不可直连第三方
- 必须实现当前城市解析接口、首页聚合接口、统计接口
- 不允许编造和风天气没有提供的第三方字段
- 统计类接口必须明确样本期与统计模式
- 后端必须负责将 `lat/lon` 解析为 `locationId`
- 后端必须负责缓存、聚合、快照落库与统计重算

## 数据库规范

- 必须使用参数化查询
- 禁止拼接 SQL
- 所有数据库相关注释必须使用中文
- 不允许假设数据库结构，必须先定义表结构和字段含义
- 数据库变更必须先同步更新表结构文档或 PRD 中的数据设计

### 当前项目的数据存储策略

- `weather_now_snapshots`、`air_now_snapshots`、`weather_alerts` 为快照型数据，按时间新增
- `weather_hourly_snapshots`、`weather_daily_snapshots`、`air_hourly_snapshots` 为预测快照，按抓取时间保存
- `weather_stats_daily`、`weather_stats_weekly`、`weather_stats_monthly` 为聚合结果，按周期更新或重算
- 月统计不是简单自增，必须基于日级数据重新聚合
- 页面展示读取“最新有效快照 + 本地聚合结果”

## 安全规范

- 禁止硬编码 API Key、数据库密码、JWT Secret 等敏感信息，必须使用环境变量
- 所有数据库查询必须使用参数化查询，防止 SQL 注入
- 所有 POST 请求必须包含 CSRF 令牌，并在服务端验证
- 所有用户输入必须进行 HTML 转义，防止 XSS
- 禁止输出敏感信息（token、密码、密钥）
- 服务端必须校验前端传入参数，不允许直接信任请求体

## 状态与性能规范

- 小型状态使用 `useState`
- 跨组件状态使用 `Zustand`
- 禁止滥用全局状态
- 列表必须使用稳定 `key`
- 大组件必须拆分
- 使用 `useMemo` / `useCallback` 做必要优化
- 路由组件支持懒加载
- 图片资源必须优化，避免无意义的大图直接进入生产环境

## 日志规范

- 错误必须记录日志
- API 请求失败必须打印错误信息
- 开发环境允许 `console`
- 生产环境必须移除无关日志

- 不允许假设数据库结构
- 不允许跳过鉴权
- 不允许省略错误处理
- 不允许生成未使用代码
- 不允许修改无关文件
- 工具失败必须有兜底策略
- 不明确需求必须先询问

## 开发工作流

### 新功能开发流程

1. 分析需求
2. 对照 PRD 确认页面、模块和接口边界
3. 若接口或字段不明确，先确认再写代码
4. 定义 TypeScript 类型
5. 设计后端接口与数据流
6. 编写组件结构
7. 编写 SCSS
8. 接入 API
9. 添加错误处理
10. 自检是否符合本文件

### 修改代码流程

1. 阅读原代码
2. 理解业务逻辑
3. 给出修改方案
4. 再进行修改

禁止直接修改代码而不分析。

### Debug 流程

1. 分析报错
2. 定位问题
3. 找到根因
4. 提供修复方案

## 输出前自检

输出前必须检查：

1. 是否全部使用 TypeScript 且无 `any`
2. 是否所有业务请求都包含 `userId`
3. 是否补全了 `loading` / `error`
4. 是否使用 `SCSS`
5. 是否符合目录结构
6. 是否存在未使用代码
7. 是否符合 ESLint 规范
8. 是否包含中文注释
9. 是否与 PRD 的页面结构和接口边界一致
10. 后端是否遵循分层、统一响应、参数校验、参数化查询与缓存约束

## 终极原则
